# P2 — EOB2: validate the shared IR on a second Westwood game — Design

> **Status:** Design (awaiting maintainer review before the per-milestone
> implementation plans are written). Opens Phase 2 (PHASES §"Phase 2 — EOB2").
> **Date:** 2026-07-09
> **Grounding:** MASTER §2 (the IR is the product — *one* IR for all four SSI
> games), §4 (importers = anti-corruption layer; two hard separations), §4.1
> (data-vs-behaviour lift, closed vocabulary — no embedded VM), §5 (IR designed
> against all four games, closed-but-extensible vocabulary, do **not** shape it
> around EOB1), §8 (differential correctness + round-trip), §10 (schema grows
> from real games, never top-down; no universal abstraction until two games
> force it); PHASES §"Phase 2 — EOB2"; **ADR-0001** (IR serialization — JSON /
> deterministic writer / fail-closed loader), **ADR-0002** (event model — closed
> `event → condition → action` vocabulary), **ADR-0003** (rendering-fidelity
> envelope).
> **Closes:** Phase 2 — one EOB2 level playable through the *same* IR/engine/
> render, proven by a diff against `kyra` EOB2 + a lossless round-trip.

---

## 1. Goal

Validate the canonical IR against a **second real Westwood game** — cheaply,
same engine family. The thesis under test (MASTER §2/§5): **one IR carries a
second game without distortion.** The proof is a real EOB2 level walked and
triggered through the *same* `beholder-ir`, `beholder-render-core`,
`beholder-engine-core`, and `beholder-ir-json` that P1 built for EOB1 — with a
second importer path the only genuinely new engine-facing surface.

Success is not "EOB2 runs"; it is "**the IR absorbed EOB2 without becoming an
EOB2 fork**." Every EOB2 construct must land in the existing IR types; where a
real EOB2 level forces a new vocabulary member, it extends the *shared* enum,
designed for the whole family (EOB1/2/3 + Dungeon Hack), never as an EOB2 patch.

## 2. Scope

**In scope** (the same "playable" depth P1 reached, on a second game):
- EOB2 geometry: walls, doors, decorations, lifted into the shared
  `LevelGeometry` / `Door` / `Decoration` / `WallArchetype`.
- EOB2 `.INF` event graph: the EOB2 trigger bytecode dialect **lifted** at import
  time into `event → condition → action` (ADR-0002), not interpreted.
- Lockstep first-person movement + collision on a real EOB2 level (reused engine).
- A lossless, byte-stable round-trip (reused `beholder-ir-json`).

**Out of scope** (YAGNI / MASTER §10 — no surface ahead of a game forcing it):
- Monsters, items (`ITEM.DAT`), combat, AD&D spell resolution, spell lists. A
  *playable EOB2 level* in P1's sense (walk + doors/decorations + one working
  trigger) does not force the monster/item/combat engine surface; deferred.
- Runtime / save-state serialization (the base-vs-runtime split's runtime half).
- Multi-level / campaign packaging (a folder of levels + shared assets).
- EOB3 / AESOP (Phase 3), Dungeon Hack (Phase 4), the editor (Phase 5).

**The load-bearing invariant — ONE IR (maintainer-emphasised):**
- **Zero EOB2-specific IR types.** No `Eob2Decoration`, no `Eob2WallArchetype`.
  Every EOB2 construct binds to the *existing* `beholder-ir` values.
- **Vocabulary extension is shared and data-driven.** If EOB2 forces a new
  `WallArchetype` / `Action` / `Condition` / `Reason` member, it grows the
  **shared** closed enum — justified by a real EOB2 level (never speculative,
  MASTER §10), and shaped coherently for the family (MASTER §5: do not distort
  the IR toward EOB1's gentleness; if a construct fits badly, that is a signal
  the IR *form* was EOB1-shaped — fix the IR, do not bend EOB2).
- **Two importers, one IR** (PHASES §Phase2). The second game legitimises
  *shared abstraction only where real duplication demonstrates it* — extracted
  from two concrete importers, never designed top-down.

## 3. Architecture

### 3.1 Importer structure — parallel EOB2 classes, shared IR (decided)
Inside the existing `beholder-importer-westwood` module, the EOB2 `.INF` decode/
lift lives as **EOB2-specific classes** (`Eob2InfScript`, `Eob2InfGeometry`,
`Eob2InfLifter`, …) **alongside** the EOB1 `Inf*` chain — both emitting only the
shared `beholder-ir` values. This is the cleanest anti-corruption layer (MASTER
§4) and the lowest risk to the green EOB1 path. A shared Westwood-`.INF`
abstraction is extracted **only where real duplication forces it** (MASTER §10 —
not ahead of need). The EOB2 `.MAZ`/CPS/VCN/VMP/palette **decoders are reused
unchanged** (see 3.2), so no separate module is warranted.

### 3.2 What is reused unchanged vs genuinely new

| Reused unchanged (proven in P1) | Confirmation |
|---|---|
| `Maze` (MAZ parser) | EOB2 `LEVEL1.MAZ` header = `width 32, height 32, nof 4`, file 4102 B — **byte-identical layout to EOB1** (the parser already validates `nof==4` + exact length) |
| CPS + Format80, VCN atlas, VMP projection, VGA palette | EOB2 ships the same Westwood formats (`CRIMSON.VCN/.VMP/.PAL`, per-area tilesets) |
| `AppearancePack` / `PixelSheet` / `DecorationTable` | framework-neutral decoded values; EOB2 fills the same shape |
| `beholder-render-core` (first-person, door/decoration occlusion) | IR-fed, game-neutral |
| `beholder-engine-core` (movement, collision, event executor) | IR-fed, game-neutral |
| `beholder-ir-json` (deterministic writer, fail-closed loader, round-trip) | IR-fed, game-neutral |

| Genuinely new (EOB2-specific) | Where |
|---|---|
| EOB2 `.INF` **geometry-decl** lift (block model, wall-mappings, decorations, doors, magic wall ids) | `Eob2InfGeometry` → shared `LevelGeometry`/`Door`/`Decoration` |
| EOB2 `.INF` **event bytecode** lift (marker-delimited script) | `Eob2InfScript` + `Eob2InfLifter` → shared event graph |
| EOB2 tileset wiring (level → VCN/VMP/palette selection) | composition root / importer |
| **Shared-vocabulary growth** where a real EOB2 level forces it | `WallArchetype` / `Action` / `Condition` / `Reason` (per-milestone, data-driven) |

### 3.3 Empirical format grounding (confirmed against the user's own data)
- **MAZ:** EOB2 `LEVEL1.MAZ` = 4102 B, header `32/32/4` — identical to EOB1;
  `Maze.parse` reuses directly.
- **INF markers + name slots:** EOB2 `LEVEL1.INF` opens with a small offset/
  pointer header, then a **`0xEC` marker** immediately followed by the 13-byte
  name slot `"level1.maz\0\0\0"` (10 chars + NUL padding to 13) and a further
  name slot (`"dun…"`). This confirms the kickoff's format facts (marker bytes
  `0xEC`/`0xEA`/`0xFF`, 13-byte name slots, block_A/B/C model, 2-byte block
  flags, different magic wall ids). The precise marker/opcode semantics are
  enumerated from the `kyra` oracle during the milestone that first needs them
  (see §4, §8), not guessed here.

## 4. The 4-milestone arc (P1-mirror, compressed)

Each milestone is individually demonstrable and differential-testable, grows the
IR from real data (never a horizontal layer ahead of use), and ends at a
maintainer checkpoint. The risk is deliberately **back-loaded**: M1–M2 prove
reuse + the new geometry decl; M3 is the genuinely-hard EOB2 script-VM lift.

Target level: **`LEVEL1`** (the entry level — parallel to EOB1's `LEVEL1`). Its
suitability is confirmed at M1's first decode; if the scripted intro makes it a
poor slice, we switch to a cleaner level then (a plan-level choice).

### M1 — EOB2 static render
Decode EOB2 assets (reused decoders) + wire the EOB2 tileset pack + reuse `Maze`
+ lift the EOB2 `.INF` **geometry declaration** (block model, wall-mappings,
decorations, doors, magic wall ids) into the shared IR → render a **headless
first-person view** of a real EOB2 level. This is the first place the shared
`WallArchetype` grows from real EOB2 data (only the archetypes `LEVEL1` actually
contains).
- **New surface:** `Eob2InfGeometry`, EOB2 tileset wiring, `WallArchetype`
  extensions *iff* forced by LEVEL1.
- **Differential evidence:** headless `int[]`/PNG golden; geometry/projection
  eyeballed against `eob2_dos.h` (dsc/door tables) + `darkmoon.cpp` (clean-room,
  read-only) + maintainer eyeball. Pixel-exact `kyra` golden deferred (no built
  ScummVM — see §6).

### M2 — walkable + doors/decorations
Reuse `beholder-engine-core` movement on the EOB2 level; door collision
(passable-iff-open) and decoration occlusion on real EOB2 data.
- **New surface:** none in the engine (proof it is game-neutral); at most EOB2
  door/decoration wiring in the importer/composition root.
- **Differential evidence:** structural walk invariants (P1 M3 shape:
  turn-cycle identity, never-off-grid, settle-at-wall, blocked-step no-op) +
  door-collision hand-verified across facings + maintainer eyeball of the walk.

### M3 — EOB2 `.INF` event lift  *(the meat / the risk)*
Lift the EOB2 marker-delimited trigger bytecode into `event → condition →
action` **at import time** (not a VM — MASTER §4.1). One real EOB2 trigger fires
end-to-end through the *shared* `beholder-engine-core` executor. Extend the
shared `Action`/`Condition`/`Reason` vocabulary **only** where a real EOB2 script
forces it.
- **New surface:** `Eob2InfScript` (marker/opcode decode + control-flow
  reconstruction) + `Eob2InfLifter`; shared event-vocabulary extensions iff
  forced.
- **Differential evidence:** the lifted trigger's effect (e.g. wall-swap / door /
  message) matches the `kyra` script behaviour for that block; the lift is
  structural (CFG → condition AST), disassembly cross-checked against the
  oracle. The three opaque named-cutscene opcodes (`sequence`/`dialogue`/
  `specialEvent`, per PHASES §3 item 3) stay opaque named actions, not decomposed.

### M4 — lossless round-trip + diff  *(closes P2)*
Reuse `beholder-ir-json`: `extract → IR_a → JSON+assets → fail-closed reload →
IR_b`, with `IR_a.equals(IR_b)`, byte-identical re-serialize, and `IR_b` renders
identically to `IR_a`; plus the diff against `kyra` EOB2.
- **New surface:** none in the codec (proof it is game-neutral); at most new
  serde only if an M1/M3 vocabulary extension introduced a new IR node kind.
- **Differential evidence:** real-`LEVEL1` round-trip IT (equality + byte-stable
  + render-identity, skip-safe, artifacts to git-ignored `target/`); optionally a
  committed synthetic EOB2-flavoured golden exercising any new node kind.

## 5. Differential-evidence & correctness posture

- **No built ScummVM** on this workstation (same as P1) → the **pixel-exact
  `kyra` golden is deferred** (documented, non-blocking). The standing oracles
  are `~/Workspaces/CPP/ScummVM/engines/kyra/engine/darkmoon.cpp` and
  `~/Workspaces/CPP/ScummVM/devtools/create_kyradat/resources/eob2_dos.h`
  (clean-room, **read-only** — never copied), plus the lossless round-trip and
  maintainer eyeball. The EOB2 `.app` bundle ships DOSBox, so a pixel ground-
  truth capture remains available as a later, separate task if wanted.
- **Round-trip** is the primary machine-checkable acceptance test (MASTER §8):
  real-level equality/byte-stability/render-identity, no game data committed.
- **Golden references** captured per level; ratchet, never regress.

## 6. Global constraints (carried from P1)

- **JDK 21** (`JAVA_HOME` = Corretto 21); `mvn spotless:apply` then `mvn verify`;
  single-module builds use `-pl <m> -am` **or** root verify (the stale `~/.m2`
  upstream-jar trap).
- **All `CODING_STANDARDS` gates green:** one `return`/method, no `instanceof`
  (visitor / `getClass()` dispatch), no `switch yield`, cyclomatic ≤ 8, NPath ≤
  120, field/method/param caps, no `return null`, `final` params, braces
  everywhere, ≤ 140-char lines, no `System.out`/`printStackTrace`, NullAway
  `ERROR`, full Javadoc, **every magic value a named `ALL_CAPS` constant + a
  what-and-why comment** (clean-room formats).
- **Coverage: JaCoCo ≥ 0.90 branch AND line** at bundle level; `model/`/records
  exempt but their guards + value-equality are behaviour-tested; new lift/decode
  logic is **not** exempt and is branch-covered by synthetic fixtures.
- **Clean-room:** `kyra` + `eoblib` are read-only oracles — never copied.
  jCodeMunch is local-only. **Never commit game data:** real-data ITs are
  `assumeTrue`-skip-safe; real-level JSON/assets/PNGs go to git-ignored
  `target/`. Synthetic goldens are our own content and ARE committed.
- **Process:** per-milestone `writing-plans` → `subagent-driven-development`
  (fresh implementer + independent review per task), TDD per task (RED → GREEN →
  commit), whole-branch review at each milestone, `--no-ff` merge on the
  maintainer's verdict; conventional branches/commits, **no worktrees**.

## 7. Open questions & risks

- **EOB2 script-VM opcode/marker set** — the exact semantics of `0xEC`/`0xEA`/
  `0xFF`, the block_A/B/C model, and the 2-byte block flags are enumerated
  (kyra-grounded) at the milestone that first needs them (geometry in M1, events
  in M3), written up as each milestone's "format facts" (the P1 rhythm) — **not
  pre-committed here**.
- **`WallArchetype` / event-vocabulary delta** — the concrete new members are
  data-driven, added at M1/M3 from what real `LEVEL1` contains. **Not enumerated
  ahead of the data** (MASTER §10).
- **Level → tileset mapping** — the `LEVEL1.INF` name slots (`0xEC` → `.maz`, and
  a further slot) suggest the tileset selection is carried in the `.INF` itself
  rather than a separate table; confirmed at M1.
- **Risk (back-loaded): the M3 script lift.** The EOB2 `.INF` is a different
  bytecode dialect from EOB1's; the lift must reconstruct control flow and map a
  new opcode set onto the shared closed vocabulary without embedding a VM. This
  is where the IR is most likely to be told it was EOB1-shaped — the intended
  stress test (MASTER §5), and the reason M3 is isolated as its own milestone.

## 8. Checkpoint (MASTER §12)

Maintainer review at the end of **each** milestone (differential evidence +
the shape of any shared-vocabulary extension), and at the **end of P2**: *did the
one IR absorb a second real game without distortion?* On approval, Phase 2 is
complete and the IR is proven against two Westwood games — the precondition for
the Phase 3 AESOP lift (the project's primary risk).
